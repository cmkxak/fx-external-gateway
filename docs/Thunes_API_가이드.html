Thunes Money Transfer V2 API 가이드

해외송금(소액해외송금) 연동을 위한 Thunes Direct Global Network REST API 정리

출처 docs.thunes.com/money-transfer/v2 · 한글 정리

1. 개요

Thunes는 전 세계 지급 생태계(모바일월렛·은행계좌·카드·현금수령)로 송금을 중계하는 글로벌 네트워크다. 우리는 이 REST/JSON API를 연동해 소액 해외송금을 처리한다.

  • 환경: Production / Pre-production (IPSec VPN 또는 HTTPS TLS 1.2+)
  • 버전: v2는 v1과 완전 분리. non-breaking 변경(신규 엔드포인트·선택필드·신규 상태)은 무공지 추가 → 클라이언트는 유연하게(미지 값 허용) 구현
  • 거래 유형: C2C(개인→개인) · C2B(개인→사업자) · B2C(사업자→개인) · B2B(사업자→사업자)
우리 소액해외송금은 C2C 기준으로 구현한다 (송금인 sender + 수취인 beneficiary, business 객체 미사용).

공통 헤더 (Headers)

API 요청·응답의 모든 헤더 필드는 대소문자를 구분하지 않는다(HTTP/1.1 RFC 7230). 따라서 헤더 이름의 대소문자에 의존하지 말 것. 응답에는 페이지네이션 정보 등 다양한 헤더가 함께 실린다.

3. 송금 플로우 (5단계)

견적으로 환율·수수료를 고정(lock)하고, 거래 생성 → 확정으로 실제 자금을 움직인다. 확정 후 상태는 콜백/폴링으로 추적한다.

① 견적 Quotation→ ② (선택) 수취인 검증 CPV→ ③ 거래 생성 Transaction→ ④ 확정 Confirm→ ⑤ 상태 Callback/Polling
단계엔드포인트설명
지급처 조회GET /payers코리도/필수필드/통화규칙 확보 → payer_id
① 견적POST /quotations환율·수수료 lock, expiration_date 부여
② 수취인 검증POST …/credit-party-verification(선택) 계좌상태/이름 검증 → 거절 예방
③ 거래 생성POST /quotations/{id}/transactionssender/beneficiary/purpose 전달 → 상태 CREATED
④ 확정POST /transactions/{id}/confirmsource 금액 hold → 망 디스패치 (자금 이동)
⑤ 상태GET /transactions/{id}콜백(권장) 또는 폴링으로 최종 상태 확인
confirm = 송금 완료 아님. hold + 디스패치까지이고, 최종 완료는 비동기(콜백/폴링)로만 확정된다. 컴플라이언스 심사도 confirm 시점에 돈다.

4. 자금 흐름 (Prefunding 모델)

Thunes는 선충전(prefunding) 방식이다. 우리가 Thunes 계좌에 미리 충전해두고, 송금 건마다 그 잔액에서 차감된다. 실시간처럼 보이지만 내부적으로는 양쪽 모두 선충전된 잔액 간 정산이다.

우리 → Thunes 충전→ Thunes 내부 (Safeguarding)→ Thunes → Payout 파트너 충전→ 수취인 지급
  • 충전 통화: USD USDC EUR GBP SGD AUD HKD IDR
  • 잔액 회계: available = balance − pending + credit_facility (confirm=hold→pending, 완료=차감, 실패=반환)
  • 잔액 소진 = 1007005(insufficient balance) = 서비스 중단 → 통화별 잔액 모니터링 필수
파트너별 계좌가 아니라 Thunes 단위 풀 계좌에서 지급. 가맹점 구분은 잔액 원장으로만. (추정 — 계약/문서 확인 권장)

출처 Thunes Money Transfer API V1 reference · 한글 정리

V1 · 개요 (Overview)

Money Transfer API는 Thunes 플랫폼을 통한 송금의 주 게이트웨이다. REST 원칙으로 구성되며 다음 기능을 제공한다: 거래 처리·조회, 계정 관리, Discovery(지급처/국가/엔티티 탐색).

환경 (Environments)

  • production / pre-production 2개 환경. IPSec VPN 또는 HTTPS로 접근.
  • 엔드포인트 상세는 계정 생성 시 별도 제공.

버전 (Versioning)

모든 엔드포인트는 버전 번호 접두사(/v1/…)를 가진다. 구현 간 완전 격리를 보장하며, major 변경이 기존 연동을 깨지 않도록 한다. major 버전 내에서는 breaking change 없음.

구분내용 (예시)
Non-breaking
(무공지 추가 가능)		신규 엔드포인트 · 기존 엔드포인트의 선택 요청필드/헤더 추가 · 응답 신규 필드/헤더 추가 · 에러 메시지 텍스트 변경 · status class 내 신규 거래상태 추가 · payer별 신규 필수필드(필수여부는 API 응답으로 판단) · 텍스트 필드 길이 증가 · 콜백 신규 필드
Breaking
(새 major 또는 사전통지)		엔드포인트 제거/이름변경 · 요청필드 허용값 변경 · (처리에 필요한 정보 외) 신규 필수 요청필드 추가 · 응답 구조 변경(필드명 변경)
클라이언트는 미지(unknown) 필드·신규 상태가 들어와도 깨지지 않게 구현해야 한다. payer별 필수필드는 하드코딩하지 말고 API 응답(required_*)으로 판단.

헤더 (Headers)

요청·응답의 모든 헤더는 대소문자 무시(HTTP/1.1 RFC 7230). 헤더 대소문자에 의존 금지. 응답에는 페이지네이션 정보 등 다양한 헤더가 실린다.

V1 vs V2 핵심 차이: ① 경로 접두사 /v1/… ↔ /v2/… · ② V1 payer는 minimum/maximum_transaction_amount·required_sender_fields·required_beneficiary_fields가 payer 최상위에 있음 (V2는 transaction_types 하위) · ③ V1엔 transaction_type(C2C 등) 개념·service 3종(1=MobileWallet, 2=BankAccount, 3=CashPickup)이 단순.

V1 · 거래 상태 & 플로우

송금이 진행되는 동안 거래는 여러 상태로 전이한다. "Upon request to customer care team"으로 표기된 전이(reversal/cancellation)는 Thunes 고객지원팀을 통해서만 수행된다.

  • 상태 변경은 콜백 URL이 제공되면 실시간 통지된다.
  • 거래 상태는 ① 반환된 id 또는 ② 송금 파트너가 부여한 external_id로 조회 가능. external_id는 응답 미수신 등 예외 시 거래를 재조회하는 우리 측 고유 참조다.
  • 각 상태의 숫자값 = status(거래 상태), 라벨 = status_class_message(상태 클래스 메시지).
CREATED→ CONFIRMED→ SUBMITTED→ AVAILABLE→ COMPLETED

Example Flow: 거래가 확정(confirm)되는 즉시 전송 주문이 수취 파트너로 전달되어 즉시 처리된다. 이 동안 송금 파트너는 CONFIRMED 상태를 받는다.
