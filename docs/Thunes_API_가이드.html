Thunes Money Transfer V2 API 가이드

해외송금(소액해외송금) 연동을 위한 Thunes Direct Global Network REST API 정리

출처 docs.thunes.com/money-transfer/v2 · 한글 정리

1. 개요

Thunes는 전 세계 지급 생태계(모바일월렛·은행계좌·카드·현금수령)로 송금을 중계하는 글로벌 네트워크다. 우리는 이 REST/JSON API를 연동해 소액 해외송금을 처리한다.

  • 환경: Production / Pre-production (IPSec VPN 또는 HTTPS TLS 1.2+)
  • 버전: v2는 v1과 완전 분리. non-breaking 변경(신규 엔드포인트·선택필드·신규 상태)은 무공지 추가 → 클라이언트는 유연하게(미지 값 허용) 구현
  • 거래 유형: C2C(개인→개인) · C2B(개인→사업자) · B2C(사업자→개인) · B2B(사업자→사업자)
우리 소액해외송금은 C2C 기준으로 구현한다 (송금인 sender + 수취인 beneficiary, business 객체 미사용).

공통 헤더 (Headers)

API 요청·응답의 모든 헤더 필드는 대소문자를 구분하지 않는다(HTTP/1.1 RFC 7230). 따라서 헤더 이름의 대소문자에 의존하지 말 것. 응답에는 페이지네이션 정보 등 다양한 헤더가 함께 실린다.

페이지네이션 응답 헤더

헤더의미
X-Total전체 레코드 수
X-Total-Pages전체 페이지 수
X-Per-Page페이지당 레코드 수
X-Page현재 페이지 번호
X-Next-Page다음 페이지 번호 (있을 때)
X-Prev-Page이전 페이지 번호 (있을 때)

커서 페이지네이션 (Balance Movements)

기타 헤더

헤더방향의미
x-simulated-transaction요청/응답시뮬레이션 거래 표시(true) — Pre-production 전용. 응답 헤더로 시뮬레이션 여부 확인 가능
우리 구현은 헤더를 대소문자 무시로 다뤄야 한다(Spring/HTTP 클라이언트는 기본적으로 case-insensitive). 페이지네이션은 응답 헤더로 내려오므로, 목록 조회(payers·balances 등) 시 바디뿐 아니라 헤더(X-Total 등)도 함께 읽어야 한다.

2. 인증

HTTP Basic Authentication — Username = api_key, Password = api_secret.

curl https://${API_ENDPOINT}/ping -u "${API_KEY}:${API_SECRET}"
  • 자격증명 = 신원·인가 그 자체. 소스코드/VCS 커밋 금지, 환경변수·Vault·VCS 제외 설정파일로 주입
  • 콜백용 키는 송금 API 키와 별도 발급

3. 송금 플로우 (5단계)

견적으로 환율·수수료를 고정(lock)하고, 거래 생성 → 확정으로 실제 자금을 움직인다. 확정 후 상태는 콜백/폴링으로 추적한다.

① 견적 Quotation→ ② (선택) 수취인 검증 CPV→ ③ 거래 생성 Transaction→ ④ 확정 Confirm→ ⑤ 상태 Callback/Polling
단계엔드포인트설명
지급처 조회GET /payers코리도/필수필드/통화규칙 확보 → payer_id
① 견적POST /quotations환율·수수료 lock, expiration_date 부여
② 수취인 검증POST …/credit-party-verification(선택) 계좌상태/이름 검증 → 거절 예방
③ 거래 생성POST /quotations/{id}/transactionssender/beneficiary/purpose 전달 → 상태 CREATED
④ 확정POST /transactions/{id}/confirmsource 금액 hold → 망 디스패치 (자금 이동)
⑤ 상태GET /transactions/{id}콜백(권장) 또는 폴링으로 최종 상태 확인
confirm = 송금 완료 아님. hold + 디스패치까지이고, 최종 완료는 비동기(콜백/폴링)로만 확정된다. 컴플라이언스 심사도 confirm 시점에 돈다.

4. 자금 흐름 (Prefunding 모델)

Thunes는 선충전(prefunding) 방식이다. 우리가 Thunes 계좌에 미리 충전해두고, 송금 건마다 그 잔액에서 차감된다. 실시간처럼 보이지만 내부적으로는 양쪽 모두 선충전된 잔액 간 정산이다.

우리 → Thunes 충전→ Thunes 내부 (Safeguarding)→ Thunes → Payout 파트너 충전→ 수취인 지급
  • 충전 통화: USD USDC EUR GBP SGD AUD HKD IDR
  • 잔액 회계: available = balance − pending + credit_facility (confirm=hold→pending, 완료=차감, 실패=반환)
  • 잔액 소진 = 1007005(insufficient balance) = 서비스 중단 → 통화별 잔액 모니터링 필수
파트너별 계좌가 아니라 Thunes 단위 풀 계좌에서 지급. 가맹점 구분은 잔액 원장으로만. (추정 — 계약/문서 확인 권장)

MethodPath설명

견적 요청 예시 (C2C, SOURCE_AMOUNT)

{
  "external_id": "9481184321482",
  "payer_id": "214",
  "mode": "SOURCE_AMOUNT",
  "transaction_type": "C2C",
  "source":      { "amount": "10", "currency": "SGD", "country_iso_code": "SGP" },

ID는 64-bit(Long) 사용 필수 (32-bit 초과). 금액은 Decimal, 통화별 precision/increment 규칙 준수.

6. 거래 상태

거시 status_class(1~7) + 미시 5자리 code. 신규 코드가 무공지 추가될 수 있으니 class 기준으로 분기하고 미시 코드는 표시/로깅용으로 둔다.

110000CREATED
220000CONFIRMED
220110CONFIRMED-UNDER-REVIEW-SLS
330000REJECTED
330200 / 30201 / 30202REJECTED-INVALID/BARRED-BENEFICIARY/SENDER
330392REJECTED-COMPLIANCE-REASON
440000CANCELLED
550000SUBMITTED
660000 / 60001/10/20COMPLETED-WITH-EXCEPTION (+ reversal 계열)

CONFIRMED-WAITING-FOR-PICKUP(취소 가능 상태)의 5자리 코드는 문서 미기재(20xxx 계열). 시뮬레이션 possible-transitions 또는 Thunes 확인 필요.

10. 요청·응답 파라미터

지급처 — Payer 객체 (GET /payers) — 실제 응답 기준

실제 응답 확인: minimum/maximum_transaction_amount 는 Payer 최상위가 아니라 transaction_types[유형] 안에 있다. 또 금액은 문자열("500000.00000000")로 내려오고, max 는 null 가능.
필드타입의미

nameString지급처(은행/월렛) 이름
precisionInteger금액 소수 자릿수 (THB=2, IDR=0)
incrementDecimal금액 최소 단위(이 배수만 허용) (THB=0.01, IDR=1)
currencyString지급 통화(ISO 4217)
country_iso_codeString지급 국가(ISO 3166-1 alpha-3)
serviceObject지급 수단 {id, name} (예: BankAccount=2)
transaction_typesObject거래유형(C2C 등)별 요구사항 (아래 표)

transaction_types[유형] 내부 (예: C2C)
